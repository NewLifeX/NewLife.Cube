# 魔方前端内置需求

## 1. 背景与目标

### 痛点

- 第三代魔方（API版 `NewLife.Cube`）已发布两年多，使用率远低于第二代 MVC 版
- 核心原因：API 版无内置前端，用户必须自行搭建 Vue/React 项目、编译、部署，门槛太高
- 魔方用户绝大部分不懂前端技术，MVC 版的核心吸引力就是"零前端开发"
- 即使是内部项目，使用前后端分离也需要同时维护 3 个项目（API后端 + 前端 + MVC后台），效率低
- StarChat 项目模式验证了：前端编译到 wwwroot 嵌入程序集，用户零前端配置即可使用

### 目标

1. 在 API 版魔方代码库中内置多套前端（Vue/React），编译产物嵌入程序集
2. 用户引用 `NewLife.Cube` + `NewLife.Cube.Vue`（或 `.React`）NuGet 包 → `dotnet run` → 浏览器打开即有完整管理后台
3. 前端通用页面根据 `GetFields` API 动态渲染，新增 Controller 和菜单后自动适配，无需额外前端开发
4. 覆盖 MVC 版 80% 功能，为全面替代做好基础


## 2. 用户角色

| 角色 | 说明 | 核心诉求 |
|------|------|---------|
| 应用开发者 | 基于魔方做二次开发的 .NET 程序员 | 引用 NuGet 即可用，不想碰前端 |
| 前端开发者 | 需要定制前端页面的前端程序员 | 能 fork 前端源码做二次开发 |
| 系统管理员 | 使用魔方管理后台进行日常操作 | 界面美观、操作流畅、功能完整 |
| 框架维护者 | NewLife 团队成员 | 降低双版本维护成本，统一代码路径 |

## 3. 功能需求

### 3.1 Vue 前端皮肤包（NewLife.Cube.Vue）
- **描述**：基于 QuickVue 迁移，提供 Vue3 + Element Plus 内置前端
- **用户故事**：作为应用开发者，我希望引用 NuGet 包后即可获得完整的 Vue 管理后台，以便零前端配置启动项目
- **验收条件**（AC）：
  - [x] 引用 `NewLife.Cube` + `NewLife.Cube.Vue` 后 `dotnet run`，浏览器访问可见登录页
  - [x] 登录后可见动态菜单、列表页、表单页
  - [x] 新增 Controller + 菜单后，前端自动渲染新页面的列表和表单
  - [x] 支持 CRUD（新增/查看/编辑/删除）操作
  - [x] 支持分页、排序、搜索功能
  - [x] 支持 OAuth 第三方登录入口
- **优先级**：Must

### 3.2 React 前端皮肤包（NewLife.Cube.React）
- **描述**：基于 CubeAntd 迁移，提供 React + Ant Design 内置前端
- **用户故事**：作为应用开发者，我希望有 React 技术栈的前端选择，以便团队按技术偏好选用
- **验收条件**（AC）：
  - [x] 引用 `NewLife.Cube` + `NewLife.Cube.React` 后 `dotnet run`，浏览器访问可见登录页
  - [x] 功能与 Vue 版对等：动态菜单、列表、表单、CRUD、分页、搜索
  - [x] 支持 OAuth 第三方登录入口
- **优先级**：Must

### 3.3 API 核心加固
- **描述**：确保 API 版提供前端渲染所需的全部接口
- **用户故事**：作为前端，我希望通过标准 API 获取菜单树、字段元数据、用户权限等信息，以便动态渲染页面
- **验收条件**（AC）：
  - [x] 菜单树接口返回完整的树形菜单结构（含图标、URL、权限标记）
  - [x] `GetFields` 接口返回完整字段元数据（类型、验证规则、下拉选项、显示控制）
  - [x] 用户信息/权限接口返回当前用户的角色和权限位
  - [x] 系统配置接口可读写 CubeSetting 等配置
  - [x] 所有 API 使用统一的 `ApiResponse<T>` / `ApiListResponse<T>` 响应格式
- **优先级**：Must

### 3.4 嵌入式静态文件服务
- **描述**：API 版支持从程序集嵌入资源提供静态文件
- **用户故事**：作为应用开发者，我希望前端资源嵌入在 NuGet 包的 DLL 中，无需手动复制静态文件
- **验收条件**（AC）：
  - [x] `CubeEmbeddedFileProvider` 在 API 核心库中可用（从 CubeNC 迁移/链接）
  - [x] 物理文件优先于嵌入资源（支持本地覆写）
  - [x] SPA 回退路由可用（非 API 请求返回 index.html）
- **优先级**：Must

### 3.5 前端二次开发支持
- **描述**：前端源码可 fork 修改，本地开发有热更新
- **用户故事**：作为前端开发者，我希望能 `pnpm dev` 启动开发服务器，修改代码实时预览
- **验收条件**（AC）：
  - [x] `web/` 目录含完整前端源码，`pnpm install && pnpm dev` 可启动开发服务器
  - [x] `pnpm build` 输出到对应 `wwwroot/` 目录
  - [x] CI/CD 可自动编译前端并打包 NuGet
- **优先级**：Should

### 3.6 文件共享重构
- **描述**：降低 CubeNC 与 Cube 双向链接文件的维护成本
- **用户故事**：作为框架维护者，我希望公共代码归属清晰，修改不会引起意外冲突
- **验收条件**（AC）：
  - [x] `CubeEmbeddedFileProvider` 统一到 API 核心
  - [x] ViewModels/DataField 等公共类型归属明确
  - [x] CubeNC 仅保留 MVC 特有的 Razor 视图和控制器覆写
- **优先级**：Should

### 3.7 文档更新
- **描述**：更新 Doc/ 文档反映新架构
- **用户故事**：作为应用开发者，我希望文档清晰说明如何使用内置前端
- **验收条件**（AC）：
  - [x] 快速开始文档涵盖 API + Vue/React 用法
  - [x] 前端二次开发文档说明 fork/build 流程
  - [x] 魔方手册目录包含新章节链接
- **优先级**：Could

## 4. 非功能需求

- **性能**：前端编译产物经 Vite 压缩 + gzip，NuGet 包体积 ≤ 15MB（单皮肤包）；首屏加载 ≤ 3s（局域网）
- **安全**：API 接口统一 JWT 认证；前端不暴露敏感配置；静态文件禁止目录浏览
- **兼容性**：.NET 6/7/8/9/10 多框架兼容；浏览器支持 Chrome/Edge/Firefox 最新 2 个主版本

## 5. 边界与约束

### 不做什么
- 不废弃 MVC 版（CubeNC），继续维护
- 不要求用户安装 Node.js（前端由 CI 编译，NuGet 包含编译产物）
- 不开发移动端专用 UI（响应式适配即可）
- 本阶段不实现所有 MVC 版特殊页面（仪表盘图表、流程图等可后续迭代）

### 已知限制
- 前端编译产物不提交 Git（CI 编译），本地开发需 Node.js 环境
- 原仓库（QuickVue/CubeAntd）保留独立维护，迁移后可能存在短期分叉
- CubeNC 与 Cube 的文件链接关系短期内保持，长期逐步整理

## 8. 术语表

| 术语 | 定义 |
|------|------|
| 第二代魔方（MVC版） | NewLife.CubeNC，NuGet 包 `NewLife.Cube.Core`，服务端渲染 |
| 第三代魔方（API版） | NewLife.Cube，NuGet 包 `NewLife.Cube`，前后端分离架构 |
| GetFields | API 接口，返回当前页面所需字段的元数据，前端据此动态渲染 |
| CubeEmbeddedFileProvider | 魔方自定义的嵌入式文件提供者，从程序集中读取静态资源 |
| IModule | 魔方模块接口，实现自动扫描加载功能插件 |
| SPA 回退 | 单页应用路由回退，非文件请求返回 index.html |
| QuickVue | 已有的 Vue3 前端项目（D:\X\NewLife.QuickVue），作为 Cube.Vue 的基础 |
| CubeAntd | 已有的 React 前端项目（D:\X\NewLife.CubeAntd），作为 Cube.React 的基础 |
