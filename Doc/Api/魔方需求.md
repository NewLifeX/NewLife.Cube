# API魔方需求（前端皮肤对接版）

> 版本：2026-04-16  
> 适用范围：NewLife.Cube 第三代（WebAPI 主线）

---

## 1. 背景与目标

当前多皮肤并行开发中，接口能力边界不一致，导致适配成本与维护成本上升。  
本需求文档用于定义“必要接口最小集 + 高级接口增强集”，统一各皮肤对接标准。

### 可衡量目标

1. 各皮肤基于必要接口可在 1 天内完成最小可运行接入。
2. 认证、菜单、CRUD 协议统一，跨皮肤行为一致。
3. 高级能力（注册、验证码、导入导出等）可按需渐进接入。
4. 新增皮肤时，重复实现成本可控，协议不分叉。

---

## 2. 用户角色

| 角色 | 说明 | 核心诉求 |
|------|------|---------|
| 平台架构师 | 负责平台能力边界与演进路线 | 接口稳定、跨皮肤一致、兼容可控 |
| 前端皮肤开发 | 负责 UI 与交互实现 | 最小接入集明确、扩展点清晰 |
| 后端开发 | 负责实体与业务接口开发 | 协议固定、减少皮肤特例 |
| 项目实施/运维 | 负责部署与日常数据管理 | 登录稳定、权限清晰、数据操作安全 |

## 3. 功能需求

### 3.1 统一协议与接口语义【必要】

- **描述**：建立 API 平台统一返回结构与路径规范。
- **用户故事**：作为前端开发，我希望所有接口遵循统一协议，以便快速接入不同皮肤。
- **验收条件（AC）**：
  - [ ] 核心接口统一返回 `ApiResponse<T>`（`code/message/data/page/stat`）。
  - [ ] 列表接口统一分页语义（`pageIndex/pageSize/totalCount`）。
  - [ ] 错误响应具备稳定 `code` 语义，前端可统一处理。
- **优先级**：Must

### 3.2 认证最小闭环【必要】

- **描述**：提供可直接上线的认证最小闭环能力。
- **用户故事**：作为前端开发，我希望仅对接最少认证接口即可完成登录态管理。
- **验收条件（AC）**：
  - [ ] 提供 `/Auth/Login`、`/Auth/Refresh`、`/Auth/Info`、`/Auth/Logout`。
  - [ ] 登录成功返回 token 对；过期后可刷新。
  - [ ] 退出登录后可清理登录态。
- **优先级**：Must

### 3.3 登录分类参数规范【必要】

- **描述**：统一登录分类参数语义，降低跨皮肤适配差异。
- **用户故事**：作为前端开发，我希望同一套登录分类约定可用于多个接口。
- **验收条件（AC）**：
  - [ ] `loginCategory` 使用字符串类型。
  - [ ] 空字符串 `""` 表示密码登录。
  - [ ] `mobile` 表示手机验证码登录。
  - [ ] `mail` 表示邮箱验证码登录。
  - [ ] `/Auth/Login` 与 `/Auth/LoginByCode` 均遵循上述约定。
- **优先级**：Must

### 3.4 菜单与权限聚合【必要】

- **描述**：提供基于 RBAC 的菜单聚合输出能力。
- **用户故事**：作为管理员，我希望不同角色组合后输出稳定可见菜单。
- **验收条件（AC）**：
  - [ ] 提供 `/Cube/MenuTree` 返回当前用户菜单树。
  - [ ] 多角色权限聚合结果稳定。
- **优先级**：Must

### 3.5 实体 CRUD 最小集【必要】

- **描述**：提供实体通用 CRUD 能力，支撑最小后台可用。
- **用户故事**：作为后端开发，我希望定义实体后快速得到标准管理接口。
- **验收条件（AC）**：
  - [ ] 支持 `GET/POST/PUT/DELETE /{type}`。
  - [ ] 支持 `GET /{type}/Detail?id=`。
  - [ ] 支持关键字 `Q`、时间区间 `dtStart/dtEnd`、排序与分页。
- **优先级**：Must

### 3.6 批量删除参数规范【高级】

- **描述**：统一批量删除 URL 参数表达，避免皮肤差异。
- **用户故事**：作为前端开发，我希望使用主流重复参数方式而非逗号拼接。
- **验收条件（AC）**：
  - [ ] 批量删除使用 `DELETE /{type}?id=1&id=2&id=3`。
  - [ ] 不使用 `id=1,2,3` 逗号拼接形式。
  - [ ] 条件删除必须至少一个过滤条件，禁止裸删全表。
- **优先级**：Should

### 3.7 安全增强（Challenge-Response）【高级】

- **描述**：提供密码传输加密机制并支持安全降级策略。
- **用户故事**：作为安全负责人，我希望降低中间链路明文泄露风险。
- **验收条件（AC）**：
  - [ ] 提供 `/Auth/Challenge` 返回 `challengeId` 与 `publicKey`。
  - [ ] `challengeId` 具备时效与一次性约束。
  - [ ] 当 `AllowPlainPassword=false` 时，未携带有效 `challengeId` 的明文登录被拒绝。
- **优先级**：Should

### 3.8 注册分类参数规范【高级】

- **描述**：统一注册分类参数语义，兼容多种注册形态。
- **用户故事**：作为前端开发，我希望注册分类与登录分类风格一致。
- **验收条件（AC）**：
  - [ ] `registerCategory` 使用字符串类型。
  - [ ] 空字符串 `""` 表示用户名密码注册。
  - [ ] `mobile` 表示手机验证码注册。
  - [ ] `mail` 表示邮箱验证码注册。
  - [ ] `oauthbind` 表示 OAuth 回跳绑定注册。
- **优先级**：Should

### 3.9 OAuth 流程分离【高级】

- **描述**：OAuth 登录与注册明确分离，避免流程耦合。
- **用户故事**：作为前端开发，我希望可清晰区分“已绑定用户登录”与“首次用户注册”。
- **验收条件（AC）**：
  - [ ] 已绑定 OAuth 用户可直接登录并返回 token。
  - [ ] 未绑定 OAuth 用户进入注册流程。
  - [ ] 注册流程通过 `/Auth/OAuthPendingInfo` + `/Auth/Register(registerCategory=oauthbind)` 完成。
- **优先级**：Should

### 3.10 登录配置与站点信息【高级】

- **描述**：提供动态登录配置与站点展示信息能力。
- **用户故事**：作为前端开发，我希望可调用接口动态渲染，但也允许硬编码兜底。
- **验收条件（AC）**：
  - [ ] 提供 `/Auth/LoginConfig`。
  - [ ] 提供 `/Cube/SiteInfo`。
  - [ ] 前端可选择调用接口或硬编码。
- **优先级**：Could

### 3.11 页面元数据与扩展能力【高级】

- **描述**：提供页面元数据与数据管理扩展能力。
- **用户故事**：作为前端开发，我希望按需接入高级能力，不阻塞最小闭环。
- **验收条件（AC）**：
  - [ ] 提供 `/{type}/GetPage` 页面元数据。
  - [ ] 提供导入导出、上传、图表接口。
  - [ ] 高级能力均不作为最小接入阻塞项。
- **优先级**：Could

## 4. 非功能需求

### 4.1 性能

- 列表接口默认分页，避免无界查询。
- 导出支持分片与上限控制。
- 高并发下认证与菜单查询保持稳定响应。

### 4.2 安全

- 支持 Challenge-Response。
- 高危删除具备参数与权限双重保护。
- Token 刷新与用户信息接口遵循 Bearer 规范。

### 4.3 兼容性

- 多皮肤共享同一后端语义。
- 支持 SPA 与 MVC 并行。
- 保持第三代 API 主线稳定。

---

## 5. 边界与约束

- 平台定位为后台快速开发 API 能力层，不绑定单一前端框架。
- 必要接口优先保障稳定，避免引入皮肤特化语义。
- 高级能力按需接入，不强制一次性全量实现。
- 文档不绑定具体前端代码实现方式。
