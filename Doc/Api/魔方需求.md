# API魔方需求

> 版本：2026-04-16  
> 适用范围：NewLife.Cube 第三代（WebAPI 主线）

---

## 1. 背景与目标

- 当前魔方已形成统一接口与多皮肤并行能力，但缺少一份面向“API平台化”的统一需求文档，导致不同皮肤、不同项目在能力边界和优先级上理解不一致。
- 我们需要把“魔方是什么、要做到什么、核心功能有哪些”明确下来，作为后续迭代、验收和兼容决策的统一依据。

可衡量目标：

1. 提供一套稳定、统一、可复用的管理后台 API 能力，覆盖 80%~90% 通用后台场景。
2. 支持多前端皮肤共享同一后端语义，核心登录、权限、CRUD、导入导出流程一致。
3. 新业务实体可通过元数据驱动快速接入，默认可用并支持局部增强。
4. 兼容存量路径（如 `/Admin/User/*`、`/Admin/Index/GetMenuTree`），新项目优先 `/Auth/*`、`/Cube/*`。

---

## 2. 用户角色

| 角色 | 说明 | 核心诉求 |
| ------ | ------ | --------- |
| 平台架构师 | 负责平台能力边界与演进路线 | 接口稳定、跨皮肤一致、兼容可控 |
| 后端开发 | 负责实体与业务接口开发 | 少写重复代码、快速交付 CRUD 与扩展能力 |
| 前端开发（多皮肤） | 负责 UI 与交互实现 | 统一协议、低适配成本、可预测行为 |
| 项目实施/运维 | 负责部署与日常数据管理 | 登录安全、权限清晰、导入导出可靠 |
| 业务管理员 | 系统最终使用者 | 菜单清晰、权限准确、操作高效 |

## 3. 功能需求

### 3.1 统一协议与接口语义

- **描述**：建立 API 平台统一返回结构与路径规范。
- **用户故事**：作为前端开发，我希望所有接口遵循统一协议，以便快速接入不同页面与皮肤。
- **验收条件**（AC）：
  - [ ] 所有核心接口统一返回 `ApiResponse<T>`（`code/message/data/page/stat`）。
  - [ ] 认证主路径统一为 `/Auth/*`，站点与公共能力统一为 `/Cube/*`。
  - [ ] 列表接口支持标准分页结构（`pageIndex/pageSize/totalCount`）。
  - [ ] 错误响应具备稳定 `code` 语义，便于前端统一处理。
- **优先级**：Must

### 3.2 认证与账号体系

- **描述**：提供可配置的统一登录、注册、找回密码与令牌能力。
- **用户故事**：作为系统用户，我希望可用密码/验证码/OAuth 登录，并在安全策略下稳定完成认证。
- **验收条件**（AC）：
  - [ ] 支持 `/Auth/Login`、`/Auth/LoginByCode`、`/Auth/Refresh`、`/Auth/Info`、`/Auth/Logout`。
  - [ ] 支持 `/Auth/SendCode`（登录/注册/重置）与 `/Auth/ResetPassword`。
  - [ ] 支持 `/Auth/Register` 与 `/Auth/OAuthPendingInfo` 的 OAuth 回跳绑定注册流程。
  - [ ] `AllowLogin/AllowRegister/EnableSms/EnableMail` 可通过配置动态控制入口可见性。
- **优先级**：Must

### 3.3 安全增强（Challenge-Response）

- **描述**：提供密码传输加密机制并支持安全降级策略。
- **用户故事**：作为安全负责人，我希望密码不以明文传输，降低中间链路泄露风险。
- **验收条件**（AC）：
  - [ ] 提供 `/Auth/Challenge` 返回 `challengeId` 与 `publicKey`。
  - [ ] 支持 RSA-OAEP/SHA-256 加密密码登录流程。
  - [ ] `challengeId` 具备时效与一次性约束，防重放。
  - [ ] 当 `AllowPlainPassword=false` 时，未携带有效 `challengeId` 的明文登录被拒绝。
- **优先级**：Must

### 3.4 权限与菜单体系

- **描述**：基于 RBAC 的“用户-角色-菜单-功能点”授权体系。
- **用户故事**：作为管理员，我希望通过角色配置菜单和动作权限，实现按权可见与按权可操作。
- **验收条件**（AC）：
  - [ ] 支持菜单树接口 `/Cube/MenuTree`，返回当前用户可访问菜单。
  - [ ] 支持动作级权限（查看/新增/编辑/删除及扩展动作）。
  - [ ] 多角色权限可聚合计算，输出稳定权限结果给前端。
  - [ ] 保留旧菜单路径 `/Admin/Index/GetMenuTree` 兼容存量系统。
- **优先级**：Must

### 3.5 实体 CRUD 与元数据驱动

- **描述**：提供实体级通用 CRUD 与字段元数据接口，支撑页面自动生成。
- **用户故事**：作为后端开发，我希望定义实体后可快速获得标准管理能力，减少手写接口。
- **验收条件**（AC）：
  - [ ] 支持 `GET/POST/PUT/DELETE /{type}` 及 `/{type}/Detail`。
  - [ ] 支持 `/{type}/GetFields?kind=` 返回列表/详情/新增/编辑/搜索字段元数据。
  - [ ] 支持关键字 `Q`、时间区间 `dtStart/dtEnd`、排序与分页查询。
  - [ ] 支持 `DataField` 的 `itemType/dataSource/url/mapField` 等渲染所需元信息。
- **优先级**：Must

### 3.6 列表页能力与批量操作

- **描述**：为管理后台高频场景提供标准列表操作能力。
- **用户故事**：作为业务管理员，我希望在列表页快速完成搜索、筛选、分页和批量处理。
- **验收条件**（AC）：
  - [ ] 支持列表分页、排序、关键字搜索与多条件过滤。
  - [ ] 支持按 ID 批量删除与按条件删除。
  - [ ] 条件删除必须具备安全保护：无过滤条件时拒绝执行。
  - [ ] 支持可选统计行返回（`stat`）用于列表汇总展示。
- **优先级**：Must

### 3.7 导入导出与数据管理

- **描述**：提供标准化数据导出、导入与高危操作安全约束。
- **用户故事**：作为实施运维，我希望按当前查询条件快速导入导出并保证数据安全。
- **验收条件**（AC）：
  - [ ] 支持 `/{type}/ExportFile?format=excel|xlsx|csv|json|xml`。
  - [ ] 支持 `/{type}/ImportFile` 上传导入。
  - [ ] 大数据量导出支持分片策略，且受 `CubeSetting.MaxExport` 限制。
  - [ ] 导出遵循当前筛选与排序条件，不允许默认整表裸导出行为。
- **优先级**：Must

### 3.8 文件上传与附件关联

- **描述**：支持独立先传与表单混传两种上传模式。
- **用户故事**：作为业务用户，我希望附件上传稳定、可回填并能与主记录自动绑定。
- **验收条件**（AC）：
  - [ ] 支持 `/{type}/UploadFile` 独立先传并返回 `attId`。
  - [ ] 支持表单提交时通过 `attachmentIds` 自动完成附件绑定。
  - [ ] 支持 multipart 混传兼容模式。
  - [ ] 上传结果返回可用于前端展示的基础信息（如路径/类型）。
- **优先级**：Should

### 3.9 图表与公共服务接口

- **描述**：提供图表数据与公共基础接口，支持首页与业务页组合。
- **用户故事**：作为前端开发，我希望无需额外拼装即可获取图表与站点公共信息。
- **验收条件**（AC）：
  - [ ] 支持 `/{type}/GetChartData` 返回标准图表配置结构。
  - [ ] 支持 `/Cube/SiteInfo` 提供站点标题、Logo、版权、登录页展示信息。
  - [ ] 支持 `/Cube/Lookup` 返回通用字典数据。
- **优先级**：Should

### 3.10 树型实体能力

- **描述**：支持树结构实体的查询与顺序调整。
- **用户故事**：作为管理员，我希望对树形数据进行完整查看与上下移动维护。
- **验收条件**（AC）：
  - [ ] 支持 `/{type}/GetAll` 返回完整树。
  - [ ] 支持 `/{type}/Up`、`/{type}/Down` 调整顺序。
- **优先级**：Could

### 3.11 多皮肤共享包与一致接入

- **描述**：通过 `@cube/*` 共享包降低多皮肤重复实现成本。
- **用户故事**：作为前端团队，我希望一套逻辑可在 Vue/React 等皮肤复用。
- **验收条件**（AC）：
  - [ ] 提供 `@cube/api-core`、`@cube/field-mapping`、`@cube/page-utils`、`@cube/auth-logic`、`@cube/page-logic`。
  - [ ] 认证流程、字段映射、分页/导出逻辑可跨皮肤共享。
  - [ ] 多皮肤登录功能矩阵保持一致（密码/验证码/OAuth/注册/忘记密码/加密传输）。
- **优先级**：Should

### 3.12 兼容性与迁移策略

- **描述**：新接口优先，旧接口保留，确保平滑迁移。
- **用户故事**：作为存量项目负责人，我希望在不一次性重构的前提下平稳迁移。
- **验收条件**（AC）：
  - [ ] 新项目默认使用 `/Auth/*` 与 `/Cube/*`。
  - [ ] 旧路径 `/Admin/User/*`、`/Admin/Index/GetMenuTree` 保持可用。
  - [ ] 文档明确“当前版本路径”与“兼容保留路径”边界。
- **优先级**：Must

## 4. 非功能需求

### 4.1 性能

- 列表接口默认分页，避免无界查询。
- 导出支持大数据分片策略，控制单次内存与响应压力。
- 高并发下认证与菜单查询应保持稳定响应。

### 4.2 安全

- 支持 Challenge-Response（RSA-OAEP/SHA-256）。
- `challengeId` 限时 + 一次性，防重放。
- 高危操作（条件删除等）必须具备参数与权限双重保护。
- Token 刷新与用户信息接口遵循 Bearer 认证规范。

### 4.3 兼容性

- 面向多前端皮肤保持统一业务语义。
- 支持 SPA 与 MVC 两类前端实现并行。
- 存量路径保留，迁移路径清晰可执行。

---

## 5. 边界与约束

- 平台定位为“后台快速开发 API 能力层”，不绑定单一前端框架。
- 平台优先解决通用后台能力，不承诺覆盖所有项目级个性化页面逻辑。
- 第二代能力以兼容维护为主，不新增重大平台特性。
- 复杂业务交互由项目侧局部定制，不在平台层强耦合实现。

## 8. 术语表

| 术语 | 定义 |
| ------ | ------ |
| API魔方 | 指 NewLife.Cube 第三代 WebAPI 平台能力层 |
| ApiResponse | 统一响应结构，包含 `code/message/data/page/stat` |
| RBAC | 基于角色的访问控制模型（用户-角色-菜单-动作） |
| Challenge-Response | 登录前获取公钥并加密密码的认证安全机制 |
| DataField | 页面元数据字段定义，用于前端动态渲染控件 |
| FieldKind | 字段模式枚举：列表/详情/新增/编辑/搜索 |
| 兼容路径 | 保留供存量系统使用的旧接口路径 |
